# 0003. 服务端 7×24 部署 + 无 idle 员工 + idea 中心

- 状态: 已接受
- 日期: 2026-07-03
- 决策者: 项目所有者

## 背景
产品要求"没有 idle 员工"——员工空闲时也在想业务、攒 idea、学技能，7×24 不间断。客户端可能随时关机，不能作为员工运行的地方。项目所有者规划：后端 / 官网 / 后台 / 客户端分处不同机器、互相连接。

## 决策
- **AI 员工(Hermes profiles)常驻后端服务器**(Docker/systemd，headless Linux daemon)。**客户端只是窗口**，关机不影响服务器上员工工作。后端/官网/后台/客户端分机部署，通过 API 互联。
- **7×24 = Hermes 原生 daemon + cron + webhooks + `/goal` + kanban dispatcher。**
- **idea 中心 = 用 cron 编排"空闲即思考/学技能"**：给员工挂定时任务(如每 30min 回顾公司近况+职责 → 产出机会/改进 idea 投 idea 中心；无 idea 则 `/learn` 补一个薄弱技能)。idea 中心与任务中心是两个东西。

## 理由
- Hermes 是服务端设计(Docker、$5 VPS 起、远程可达)，天然适合常驻后端。
- Hermes 的学习循环是**每轮对话后触发**，**不会自发"发呆时冒想法"**；所以"空闲思考"需要我们用 cron 主动驱动。
- 定位诚实：这是"编排出来的勤奋"，不是自发意识，但产品体验足够。
- 详见 [../ARCHITECTURE.md](../ARCHITECTURE.md) §3.6。

## 后果
- 需要一个 idea 中心的数据模型 + 展示，独立于任务中心。
- 多员工 = 多常驻进程，服务器 RAM/CPU 与 LLM 速率/成本是约束，需要规模策略。
- cron 频率、idea 去重/质量筛选需要设计，避免刷屏。
