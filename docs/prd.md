# AgentPulse 一人自媒体公司 MVP PRD

## 1. 文档目的

本文定义 AgentPulse 第一阶段“一人自媒体公司”MVP 的产品范围、用户场景、端到端流程、功能优先级、成功标准和延期边界，作为后续 agent-system、architecture、UX、研发拆分和 QA 验收的共同上游。

第一阶段目标不是做一个全行业通用 AI 自动化平台，而是跑通一个普通用户能理解、能管控、能验收的闭环：

```text
创建内容工作室 -> 获得默认 AI 员工 -> 输入内容经营目标 -> 系统拆解任务
-> 多员工协作执行 -> 展示进度和产出 -> 高风险动作等待用户确认
```

## 2. 背景和问题

AgentPulse 面向普通人和一人公司。用户不是来和单个聊天机器人反复对话，而是希望像搭建一家小公司一样，把业务目标交给一支 AI 团队，并能看见每个 AI 员工在做什么、为什么这么做、产出了什么，以及哪些动作需要自己确认。

第一阶段选择“一人自媒体公司”作为 MVP，是因为它有清晰的高频任务：

- 选题和内容计划。
- 草稿、标题、脚本、发布说明。
- 资料、品牌语气和历史内容复用。
- 发布、保存、外部发送等高风险动作需要用户确认。

## 3. 目标用户

### 3.1 核心用户

一人自媒体公司经营者，通常具备以下特征：

- 独立创作者、知识博主、咨询顾问、个体创业者或小团队负责人。
- 同时承担老板、策划、执行、复盘等角色。
- 有明确内容经营目标，但缺少稳定的内容生产流程。
- 希望 AI 帮自己拆解任务和生成产出，但不希望 AI 擅自发布、发送或改动重要文件。
- 对 agent、run、workflow、tool 等技术概念不熟悉，更容易理解“公司、员工、任务、资料库、审批、产出”。

### 3.2 非核心用户

以下用户不作为第一阶段优先服务对象：

- 多部门企业团队。
- 需要复杂 CRM、客服工单、财务流程的企业用户。
- 需要跨平台自动发布和全量 SaaS 集成的增长团队。
- 希望自己搭建通用 workflow builder 的高级自动化用户。

## 4. MVP 目标

### 4.1 产品目标

- 让用户能创建或看到一个“一人自媒体公司”的工作区。
- 让用户理解系统默认提供 3 个 AI 员工，并知道每个员工负责什么。
- 让用户输入一个内容经营目标后，系统能自动拆解为可追踪任务。
- 让用户看到每个任务的负责人、状态、执行日志和产出。
- 让高风险外部动作进入“待确认”状态，而不是自动执行。

### 4.2 业务目标

- 验证普通用户是否理解“AI 公司工作台”这个产品表达。
- 验证“一人自媒体公司”是否能形成可演示、可复现的端到端闭环。
- 验证默认 AI 员工和任务拆解是否比单轮聊天更容易带来掌控感。

## 5. 非目标

第一阶段明确不做：

- CRM、财务、客服、销售自动化和跨行业模板。
- 多用户企业权限、团队空间和复杂角色权限。
- 真实发布平台接入，例如小红书、公众号、抖音、邮件群发。
- 自动对外发送、自动发布、自动评论、自动私信。
- 通用 workflow builder、复杂 DAG 编排和触发器自动化。
- 计费、套餐、额度、支付和商业化后台。
- 全量资料库索引、企业级 connector marketplace。
- 直接验收当前非 `docs/` 前端改动。

## 6. 用户故事

### Story 1：创建内容工作室

作为一人自媒体创作者，我希望创建一个“内容工作室”，并自动获得老板助理、内容策划、运营执行 3 个 AI 员工，以便我不用先理解技术配置，也能开始把内容经营目标交给 AI 团队。

验收标准：

- 用户能看到公司工作区名称、目标平台和账号定位。
- 用户能看到 3 个默认 AI 员工及其职责说明。
- 用户不需要手动配置 prompt、模型、工具权限即可开始第一个任务。

### Story 2：输入内容目标并自动拆解任务

作为一人自媒体创作者，我希望输入“为下周小红书账号生成一周内容计划”这样的目标后，系统自动拆成策划、撰写、整理和确认任务，以便我能从目标直接进入可执行流程。

验收标准：

- 用户能提交一段自然语言目标。
- 系统能生成主任务和若干子任务。
- 每个子任务有负责人、状态、预期产出和下一步。

### Story 3：查看 AI 员工协作进度

作为用户，我希望看到每个 AI 员工正在处理什么、用了哪些资料、产出了什么内容，以便我判断过程是否可靠，而不是只等待最终答案。

验收标准：

- 任务详情中展示 timeline 或日志列表。
- 日志使用普通用户语言，不暴露 chain-of-thought。
- 日志至少包括任务创建、任务分配、执行开始、产出生成、等待确认、任务完成等关键事件。

### Story 4：确认高风险动作

作为用户，我希望 AI 在保存重要文件、写入长期资料库、准备外部发布或发送前先让我确认，以便我能控制对外可见内容和重要数据变更。

验收标准：

- 高风险动作显示为待确认请求。
- 确认请求包含执行者、动作类型、目标对象、变更摘要、风险说明和用户选择。
- 用户至少可以选择批准、拒绝、修改后再执行或只保存为草稿。
- MVP 不真实执行外部发布或发送，只生成待确认草稿。

## 7. 端到端流程

### 流程：生成小红书账号一周内容计划

1. 用户进入桌面工作台，看到一个默认工作区：`小红书内容工作室`。
2. 系统展示 3 个默认 AI 员工：
   - 老板助理：理解目标、拆解任务、汇总结果、发起确认。
   - 内容策划：生成选题、内容结构、标题和素材建议。
   - 运营执行：整理发布计划、草稿、表格和 Markdown 产出。
3. 用户输入目标：`帮我为下周的小红书账号做一周内容计划，账号定位是 AI 工具效率提升，目标是吸引自由职业者。`
4. 老板助理创建主任务，并拆解为子任务：
   - 明确账号定位、受众和目标。
   - 生成 7 天选题方向。
   - 为每条内容生成标题、角度、脚本大纲和素材建议。
   - 整理为一周发布日历和 Markdown 草稿。
   - 请求用户确认是否保存为本地文件或作为待发布草稿。
5. 内容策划接收策划任务，读取品牌资料和历史内容摘要，生成选题列表。
6. 运营执行接收整理任务，把选题和草稿整理为表格与 Markdown。
7. 系统展示执行日志和产出：
   - 哪个员工处理了哪个任务。
   - 当前状态是进行中、等待用户确认还是已完成。
   - 产出内容在任务详情页可预览。
8. 当运营执行准备保存文件或准备发布时，系统进入 `waiting_user`：
   - 显示目标文件名或发布平台。
   - 显示变更摘要和风险说明。
   - 用户选择“只保存为草稿”。
9. 系统完成任务，并把内容计划归档到任务记录中。

## 8. 默认 AI 员工

| 员工     | 职责                         | 第一阶段允许做                                    | 第一阶段禁止做                         |
| -------- | ---------------------------- | ------------------------------------------------- | -------------------------------------- |
| 老板助理 | 目标澄清、任务拆解、结果汇总 | 创建任务、分配任务、汇总进度、提出确认请求        | 自动发布、自动发送、绕过用户确认       |
| 内容策划 | 选题、结构、标题、内容方向   | 读取允许的品牌资料、生成选题、生成内容大纲        | 修改重要文件、写入私密资料、直接发布   |
| 运营执行 | 整理草稿、发布计划、导出准备 | 整理 Markdown、表格、发布说明、生成待确认保存请求 | 真实发布到外部平台、群发邮件、删除资料 |

第一阶段不新增独立 Reviewer 员工。质量检查由老板助理在汇总阶段承担，避免角色过多导致用户难以理解。

## 10. 状态和确认节点

任务状态采用第一阶段简单状态机：

```text
queued -> assigned -> running -> waiting_user -> completed
                         |              |
                         v              v
                       failed        cancelled
```

必须进入用户确认的动作：

- 发布或发送到外部平台。
- 发送邮件、私信、评论或客服回复。
- 保存、覆盖、删除本地重要文件。
- 写入长期资料库或更新品牌记忆。
- 使用私密资料或外部账号凭证。
- 产生费用、连接付费工具或修改权限。

确认请求必须展示：

- 动作类型。
- 发起 AI 员工。
- 目标对象。
- 变更摘要。
- 风险说明。
- 用户可选动作：批准、拒绝、修改后再执行、只保存为草稿。

MVP 中，外部写入类动作不执行真实外部调用，只生成待确认草稿和说明。

## 11. 成功标准

### 11.1 产品成功标准

- 新用户能在 5 分钟内理解 AgentPulse 是“一人自媒体公司”的 AI 工作台，而不是普通聊天机器人。
- 用户能完成一次从内容目标输入到内容计划产出的完整流程。
- 用户能说清 3 个默认 AI 员工分别负责什么。
- 用户能在任务详情中看见进度、负责人、状态和产出。
- 用户能在高风险动作前看到确认请求，并拒绝或改为草稿。

### 11.2 工程和验收标准

- 后续研发 issue 能从本 PRD 拆出清晰范围、依赖和验收标准。
- QA 能复现至少一条端到端手动流程。
- 所有对外发布、发送、删除、覆盖、授权、计费动作都有确认节点或明确延期。
- MVP 不因引入 CRM、财务、客服、跨行业模板而扩大范围。

## 12. 延期项和原因

| 延期项             | 延期原因                                   | 可重新评估时机                 |
| ------------------ | ------------------------------------------ | ------------------------------ |
| 自定义 AI 员工     | 默认 3 角色尚未验证，过早开放会增加复杂度  | 默认角色闭环通过后             |
| 真实发布平台接入   | 涉及外部账号授权、内容合规和平台 API 风险  | 确认节点和草稿流稳定后         |
| CRM / 客服 / 财务  | 偏离一人自媒体公司内容生产主线             | 第二个垂直场景明确后           |
| 企业多用户权限     | 第一阶段是单人工作台，不需要团队权限模型   | 出现多成员协作需求后           |
| 通用 workflow 编排 | 普通用户理解成本高，且会拖慢 MVP 验证      | 多个固定流程跑通并沉淀模板后   |
| 全量资料库索引     | 数据权限、检索质量和本地性能都需单独设计   | 资料库占位和少量文本资料验证后 |
| 计费和额度系统     | 当前目标是验证产品闭环，不是商业化收费流程 | MVP 演示和早期用户验证完成后   |

## 14. 假设、风险和待验证项

### 14.1 当前假设

- 用户愿意用“公司 / 员工 / 任务 / 审批”的方式理解 AI 协作。
- 一人自媒体公司是足够聚焦且高频的 MVP 场景。
- 默认 3 个 AI 员工足以覆盖第一阶段内容计划闭环。
- 任务拆解和进度展示可以先用 mock 或规则化逻辑验证体验，再接入真实 runtime。

### 14.2 主要风险

- 范围膨胀：过早加入 CRM、客服、财务、跨行业模板。
- 能力误解：用户以为系统已经能真实发布或连接外部账号。
- 过程不透明：如果只展示最终答案，会退化为普通聊天机器人。
- 确认节点缺失：发布、发送、覆盖文件等动作若没有确认，会损害信任。
- 架构过重：过早实现复杂权限、connector 和 workflow builder，会拖慢 MVP。

### 14.3 待验证项

- 用户是否能理解 3 个默认 AI 员工的分工。
- 用户是否觉得任务 timeline 比聊天记录更有掌控感。
- “只保存为草稿”的确认动作是否足够降低外部发布风险。
- 内容计划产出的格式是否应优先是 Markdown、表格，还是两者都需要。
- 资料库第一阶段需要支持哪些最小字段，才能让内容策划显得有上下文。
